

# 4hr RPO (Recovery Point Objective) 
Definition: The maximum allowable amount of data loss, measured in time, that a business can tolerate. It is the maximum age of files or data in backup storage required to resume normal operations.
What it means for "4hr": If a system fails at 12:00 PM, the last valid backup must have been taken no later than 8:00 AM (4 hours prior).
Impact: This dictates that backups or data replications must occur at least every 4 hours. If a disaster occurs, you may lose up to 4 hours of data, but no more. 


# 12hr RTO (Recovery Time Objective)
Definition: The maximum tolerable amount of time a system, application, or business process can be down after a disaster, failure, or disruption.
What it means for "12hr": If a system goes down at 12:00 PM, it must be fully restored, tested, and operational by 12:00 AM (12 hours later).
Impact: This defines the speed at which IT staff must restore data from backups and bring systems back online. 

Metric   	 What it Measures	     4hr RPO / 12hr RTO Example
RPO (4hr)	Data Loss Tolerance	  You can lose, at most, 4 hours of data.
RTO (12hr)	Downtime Tolerance	  You must be back in business within 12 hours.


# Real-World Example
Imagine a company has a critical customer-facing database.
At 10:00 AM, a ransomware attack shuts down the system.
4hr RPO: The company must have a backup from at least 6:00 AM. If the last backup was at 5:00 AM, the 4-hour requirement is violated (too much data lost).
12hr RTO: The IT team must have the system fully restored and functional by 10:00 PM (12 hours later). If it takes 13 hours to restore, the RTO is violated (too much downtime). 


# How do you collect the logs in ELK stack from Kubernetes pods?
In a Kubernetes environment, logs from pods can be collected and sent to the ELK (Elasticsearch, Logstash, Kibana) stack using a log shipper like Fluentd or Filebeat. Here’s a high-level overview of how this works:
1. Deploy a Log Shipper: Install Fluentd or Filebeat as a DaemonSet in your Kubernetes cluster. This ensures that a log shipper runs on each node, collecting logs from all pods on that node.
2. Configure Log Collection: Configure the log shipper to tail the log files generated by the containers. In Kubernetes, container logs are typically stored in the /var/log/containers/ directory on each node.
3. Parse and Transform Logs: Set up parsing rules in the log shipper to structure the logs appropriately. This may involve extracting fields like timestamp, log level, message, pod name, namespace, etc.
4. Forward Logs to ELK: Configure the log shipper to forward the collected and parsed logs to Logstash or directly to Elasticsearch. This usually involves specifying the Elasticsearch endpoint and any necessary authentication.
5. Visualize in Kibana: Once the logs are in Elasticsearch, you can use Kibana to create visualizations, dashboards, and perform searches on the log data.
Example Fluentd Configuration Snippet:
<source>
  @type tail
  path /var/log/containers/*.log
  pos_file /var/log/fluentd-containers.log.pos
  tag kubernetes.*
  format json
Fluentd is an open-source, vendor-neutral data collector that acts as a unified logging layer, enabling you to collect, transform, and ship logs from various sources to multiple destinations (e.g., Elasticsearch, S3, Kafka). It structures data as JSON, making it ideal for microservices and cloud-native environments like Kubernetes. 



1. IOPS (Input/Output Operations Per Second) 
  The number of read and write operations a storage device performs in one second.
  Use Case: A key metric for assessing the speed of storage devices, particularly for random, small-block operations common in databases.

2. Throughput (Data Transfer Rate)
  The amount of data that can be transferred to and from a storage device in a given time, typically measured in megabytes per second (MB/s) or gigabytes per second (GB/s).
  Use Case: Important for applications that handle large files or sequential data access, such as video streaming or big data processing.

3. Latency (Response Time)
 The time taken for a single I/O request to be completed, measured from the moment it is sent until the acknowledgement is received.
 Use Case: A key indicator of performance degradation. High latency usually means the storage system is overloaded or experiencing issues.
